## 📋 Original Requirements (JIRA Story)

### Story: Deleting MedGo Account

**Description:**
As a MedGo user, I want to delete my app account while preserving my medical records in MediCard's central database, ensuring compliance and data retention policies are met.

### Acceptance Criteria:

#### 1. Access & Initiation
- **Settings Screen:**
  - Under "Others" or "Account", a "Delete Account" button is visible (e.g., red text for emphasis).
  - Tapping it opens a confirmation flow.

#### 2. Account Deletion Flow
- **Password Verification:**
  - Requires the user to enter their current password.
  - If incorrect: Shows error "Incorrect password. Try again." (max 3 attempts).
  
- **Final Confirmation:**
  - Displays a clear warning:
    "Deleting your account will remove your access to MedGo. Your medical records will be retained by MediCard for legal and operational purposes as per our data retention policy. This action cannot be undone."
  - Requires explicit consent (e.g., checkbox: "I understand this action is permanent").

#### 3. Deletion Process
- **On Successful Verification:**
  - Removes the user's account from the MedGo app database (auth credentials, app preferences, etc.).
  - Preserves all medical records (LOAs, reimbursements, appointments) in MediCard's central database.
  
- **Confirmation:**
  - Shows: "Your account has been deleted. Thank you for using MedGo."
  - Redirects to the login screen within 3 seconds.

#### 4. Data Handling
- **Medical Records:**
  - All historical data (claims, visits, prescriptions) remains intact in MediCard's secure records system.
  
- **App-Specific Data:**
  - Only app-related data (login, settings, activity logs) is deleted.

#### 5. Edge Cases
- **Network Issues:**
  - "Deletion failed due to connection issues. [Retry] when online."
  
- **Pending Transactions:**
  - If reimbursements/LOAs are in progress, warns: "Complete pending actions before deletion."

## 🔐 How It Works

### 1. User Identification Logic

The API automatically determines if `userId` is an email or memberCode:

```java
// Email detection: contains "@" and "."
boolean isEmail = userId.contains("@") && userId.contains(".");

if (isEmail) {
    deletionBasis = "EMAIL";
    // Search by email in both member and non-member tables
} else {
    deletionBasis = "MEMBER_CODE";
    // Search by memberCode in member table only
}
```

### 2. Password Verification Flow

1. **Check if account is locked:**
   - If `DELETE_ACCOUNT_LOCK_TIME` is in the future → Account is locked
   - Return error with remaining lock time

2. **Get attempt count from database:**
   - Read `FAILED_DELETE_ACCOUNT_ATTEMPTS` from user record
   - If lock expired, reset attempts to 0

3. **Verify password:**
   - If password is incorrect:
     - Increment `FAILED_DELETE_ACCOUNT_ATTEMPTS`
     - Save to database
     - Return error with remaining attempts
   - If password is correct:
     - Reset `FAILED_DELETE_ACCOUNT_ATTEMPTS` to 0
     - Clear `DELETE_ACCOUNT_LOCK_TIME`
     - Continue with deletion

4. **Max attempts exceeded:**
   - If attempts >= 3:
     - Set `DELETE_ACCOUNT_LOCK_TIME` = current time + 1 minute
     - Save to database
     - Return error

### 3. Deletion Process

1. **Check for pending transactions** (members only):
   - Placeholder for LOA/Reimbursement service integration
   - If pending transactions exist → Block deletion

2. **Delete app-specific data:**
   - Delete trusted devices from `MEDGO_TRUSTED_DEVICES`
   - Delete user account from `MEDGO_USERS` (members) or `MEDGO_NON_MEMBER_USER` (non-members)

3. **Preserve medical records:**
   - Medical records remain in MediCard's central database
   - No deletion from `claims_portal` or membership tables

### 4. Response Format

The response message dynamically includes the deletion basis:

```java
String deletionBasisLabel = deletionBasis.equals("MEMBER_CODE") ? "memberCode" : "email";
String successMessage = String.format(
    "Your account has been deleted. Thank you for using MedGo.%s : %s", 
    deletionBasisLabel, 
    deletedIdentifier
);
```

## 🔄 Member vs Non-Member Handling

### Member Users:
- **Table:** `MEDGO_USERS`
- **Can be deleted by:** Email OR MemberCode
- **Pending transactions check:** Yes (if memberCode exists)
- **Medical records:** Preserved in MediCard central DB

### Non-Member Users:
- **Table:** `MEDGO_NON_MEMBER_USER`
- **Can be deleted by:** Email only (no memberCode)
- **Pending transactions check:** No
- **Medical records:** N/A (no medical records)

---

## 🚨 Error Handling

### Error Scenarios:

1. **User Not Found (404)**
   - User doesn't exist in either table

2. **Invalid Password (401)**
   - Password doesn't match
   - Returns remaining attempts

3. **Max Attempts Exceeded (429)**
   - 3 failed attempts
   - Account locked for 1 minute

4. **Inactive User (403)**
   - User status is not ACTIVE

5. **Confirmation Not Provided (400)**
   - `confirmed` field is false or null

6. **Pending Transactions (409)**
   - Member has pending LOAs/reimbursements
   - Deletion blocked

7. **Network Error (500)**
   - Connection issues during deletion
